Executive Briefing

把 2000 行表格变轻，不把产品行为变险

这份汇报把 DataTable 虚拟滚动计划翻译成一页能直接讲给产品、架构、前端和测试听懂的材料。 它不是只讲“怎么快”，而是讲“怎么快、怎么稳、怎么随时撤退”。

2000 → ≤ 64 目标是在每页 2000 条时，把 DOM 行规模压到 64 个 tr 以内。
5 个任务 先做 Task 0 方向 gate，再做基线、分层、核心实现，最后用 Playwright 做总回归。
3 层降级 配置开关、浏览器守卫、运行时 fallback，确保线上异常时能局部撤退。
一句话结论 这次不是把表格“重写成新组件”，而是在保留现有 DataTable、TanStack Table、React Query、 workspace tabs 结构不变的前提下，只在渲染层加一个“看得到多少，就画多少”的调度能力， 并把统一行高、单行截断和增强预览一起收进共享层。
汇报对象 架构评审看运行时拓扑和回滚路径。前端执行者看 task 顺序和验证标准。产品与测试看可观测性、 行为不变性、上线边界。

一分钟摘要

如果只拿 3 个结论去开会，记这 3 个。

为什么要做

问题不是查询慢，是当前页画太多

产品表每页能开到 500 / 2000。数据已经分页，但当前页的每一行仍然被完整渲染进 DOM。 这会拖累滚动、布局、重绘和调试体验。

怎么做

保留原生 table，只虚拟 body

不改成 grid，不碰 `useDataTable`，不改变 React Query 的查询链路。 首版先守住当前 `absolute-position virtual rows` 原型，只把 `DataTableBody` 收敛成“当前页只保留可见行”的渲染层策略， 同时给管理后台一套统一 compact row 规范，并用 Task 0 决定是否还值得继续评估 spacer row。

怎么保命

任何一步不成立，立刻停

行高不稳定、sticky / pinned 冲突、Suspense remount 时序不稳、辅助技术语义不对， 任一命中就停止 fixed-height 方案，回写新的 `measureElement` 计划。

一个寓言，先讲给非前端听

比喻只是降低理解门槛，后面每一段都给出精确的工程翻译。

港口与货箱

想象产品表是一个仓库，仓库里有 2000 个货箱。以前的做法是： 客户只看门口 20 个货箱，我们也先把 2000 个都摆进展厅。结果不是货不够，是展厅被挤爆。

虚拟滚动像一个聪明的仓库管理员。管理员不再一次性把所有货箱都搬进来，而是只把门口视野内的那一小段货箱摆出来， 再按刻度把这些货箱挂到它们在长走廊中的正确高度。

但这个管理员不能只会搬货。客户翻页时，他要先把门口清零；仓库临时拉起隔离带时， 他要记得回来后从头开始；仓库关灯保活时，他不能在黑暗里继续瞎算；如果管理员自己摔倒了， 仓库也不能关门，只能回到“全部摆出来”的保守模式。

工程翻译 DataTable 只虚拟 ``，V1 先保留当前 absolute rows 定位策略； pagination / sorting / filters 变化时 pre-paint reset； `Suspense` remount 时首帧从 index 0 重建； keepAlive hidden 时冻结 observer； 文本列统一单行截断，并用原生 `title` / `aria-label` 暴露完整内容； runtime error 局部降级成完整渲染。

技术路线

这部分讲“为什么选这条路”，而不是只给实现细节。

问题建模

现状里的 4 个真实约束

  • 1. React Query + Suspense 会导致完整 unmount / remount 产品表不是简单 props 更新，翻页时可能先进入 fallback，再重新 mount。
  • 2. workspace V2 有 keepAlive hidden 生命周期 离开 products tab 并不等于组件销毁，observer 会遇到 0x0 viewport。
  • 3. 右侧 actions 列已使用 sticky pinning 所以虚拟方案不能随意改布局模型，否则会直接打坏对齐。
  • 4. 这是共享 DataTable，不是 products 专属组件 users 表等调用方必须继续保持非虚拟化路径和现有行为。
选型结论

为什么首版先守住 absolute rows

  • 保守 不把原生 `` 改成 grid / flex，降低 sticky header 和 pinned columns 的结构性风险。
  • 局部 仅改 `src/components/ui/table/data-table*.tsx` 渲染层，不把 `react-virtual` 泄漏进 `useDataTable`。
  • 统一 把默认行高、垂直 padding、截断策略和原生 `title` / `aria-label` 做成共享 preset，不让每个列表页自己猜像素值。
  • 可回退 若 Task 0 证明 spacer row 不稳或 ROI 不足，V1 直接守住 absolute rows；若 fixed-height gate 失败，再切换到新的 `measureElement` 方案。
    • 架构图

    运行时数据流

    Step 1 用户翻页 / 排序 / 筛选
    Step 2 `table.getState()` 更新
    Step 3 feature 层同步为 API filters
    Step 4 `useSuspenseQuery` 触发 fallback / remount
    Step 5 新 rows 到达，virtual range 从 0 重建
    关键点不是“能不能渲染出来”，而是： scroll reset 必须发生在 query 同步 effect 之前， 否则首帧会先出现旧滚动位置的旧 range，再跳回顶部。

    为什么不是其他方案

    这是评审会上最常见的问题，提前给出回答。

    方案 优点 缺点 结论
    继续完整渲染 最简单、零新逻辑 2000 行 DOM 成本太高，问题继续存在 不接受
    改成 grid / flex 虚拟表格 布局自由、虚拟实现成熟 会直接改变现有表头、sticky、pinning 和原生 table 语义 首版不做
    absolute-position virtual rows 与现有原型一致，列宽与 pinning 风险已知，便于先做 ROI 验证 仍需处理 remount、a11y、hidden viewport 与局部降级 V1 主线
    固定行高 + spacer rows 更接近正常文档流，理论上更利于部分语义对齐 列宽、sticky、pinning 与 table-layout 协商都需要额外 gate 证明 仅作候选路线
    `measureElement` 动态测量 能兼容可变行高 实现与观察成本更高，keepAlive / hidden 场景更复杂 作为失败后的 replan 方案

实现方案

按执行顺序拆成 5 个任务，每一步都有明确验证出口。

T0
方向 Gate

先做 Profiling 与 Column Width Stability Gate

第一件事不是继续写虚拟化代码，而是先证明这件事值不值得做，以及 spacer row 会不会打坏列宽、sticky 和 pinning。

  • 拆开看首屏/翻页提交耗时与滚动过程 scripting/layout/paint。
  • 记录 DOM virtualization 能解决和不能解决的瓶颈。
  • 30 分钟 feasibility check 验证 spacer row 是否过列宽稳定性 gate。
T1
基线

建立 DataTable 基线测试与浏览器预检

先证明现在的产品表 viewport selector 稳定可寻址，并记录每页 2000 条时， `tbody` 确实渲染了 2000 行，作为后面 DOM 降幅的对照基线。

  • 补 `data-table.test.tsx` 的真实 harness。
  • 补 rollback Playwright smoke。
  • 记录 `2000 rows` 的未虚拟化基线。
T2
分层

收敛现有 body 原型并暴露 viewport ref

仓库里已经有一版 `data-table-body.tsx` prototype。第二步不是新建，而是先把它收敛成干净的渲染分层基线， 同时在 `ScrollArea` 上暴露安全的 viewport ref，并把 compact row 视觉规范收进共享配置。

  • `DataTable` 只保留 orchestration。
  • `ScrollArea` ref API 只做 additive 扩展。
  • 提前审计 sticky containing block 风险。
T3
核心

完成虚拟滚动、时序契约、a11y 与降级策略

这是计划最重的一步。基于当前 absolute rows 路线完成 virtual rows、pre-paint reset、 `Suspense` remount、keepAlive hidden guard、telemetry、products-only gate 与运行时降级。

  • 固定行高 gate 不通过就停止，不硬调魔法数。
  • 文本列首版只做 `ellipsis + title + aria-label`，不引入新 preview 子系统。
  • 运行时错误只局部降级，不打崩整页。
T4
总验收

用 Playwright 做跨任务回归

最终不是“肉眼看着差不多”，而是用 `data-virtual-*`、`data-row-index` 和 `getBoundingClientRect()` 做可证伪断言。

  • 验证 tab 切换前后不 blank、不乱跳。
  • 验证 pinned actions 列在 absolute rows 路线下始终贴齐。
  • 验证 `tbody tr <= 64` 的 DOM budget。

关键实现点，通俗翻译

这部分适合给非前端或新接手工程师快速建立脑图。

A11y

不要让读屏器丢掉真实行号

首版主线没有 spacer rows，但仍然只有一部分真实数据行留在 DOM 里。 所以必须让辅助技术知道“总共有多少行”，以及“当前读到的是第几行”。

真正的 virtual row 要显式写 `aria-rowindex`，表格总数要通过 `aria-rowcount` 告诉辅助技术。

Suspense

补货车换班时，门口要先清零

切页会触发 `useSuspenseQuery`，表格可能先卸载再重挂。 所以不能靠“上一个实例记得滚到顶部”碰运气，新实例首帧必须从 index 0 开始。

KeepAlive

仓库关灯时，不要继续瞎点货

workspace tabs 会把页面挂成 `Activity hidden`。这时 viewport 可能变成 0x0。 virtualizer 必须暂停观察或冻结 snapshot，避免后台无意义重算。

统一视觉策略

你要的“看起来整齐”，在工程上不能靠约定俗成，必须落成共享 preset。

Preset

共享 compact row token

默认行高、cell 垂直 padding、overscan、阈值统一放进共享配置。页面只表达“我要启用”，不重复硬写 `56/8/100`。

Text

单行截断，不放开多行

虚拟化首发范围内，文本列统一 `nowrap + ellipsis`。这样视觉整齐、行高稳定、测量面最小。

Preview

V1 先用原生 title

首版不为了单个长文本列引入新的 preview / popover 子系统。完整内容先通过原生 `title` 与必要的 `aria-label` 暴露，后续再按 ROI 独立迭代。

使用示例

给执行者看的是“怎么接”，给评审看的是“接上以后会发生什么”。

示例 1

产品表启用方式

return (
  <DataTable
    table={table}
    scrollTargetId="products-table"
    virtualization={true}
  >
    <DataTableToolbar table={table} />
  </DataTable>
)
核心思想：products 表是首发试点。users 表先不启用。开启体验尽量接近布尔开关，默认 preset 由共享配置层托管。
示例 2

共享 compact-admin preset

export const productTableVirtualization = {
  gate: import.meta.env.VITE_ENABLE_PRODUCT_TABLE_VIRTUALIZATION !== 'false',
  compactRowHeight: 56,
  compactCellPaddingY: 10,
  overscan: 8,
  rowCountThreshold: 100,
}
`56` 在这里的角色是共享设计 token，不是每个 feature 自己维护的魔法数。
示例 3

文本列首版做法

cell: ({ cell }) => {
  const value = cell.getValue<string>()
  return (
    <div className="truncate" title={value} aria-label={value}>
      {value}
    </div>
  )
}
首版目标是统一行高和完整文本可达，不在虚拟化主线上引入新的共享预览组件。
示例 4

products-only gate

export const productTableFlags = {
  enabled: import.meta.env.VITE_ENABLE_PRODUCT_TABLE_VIRTUALIZATION !== 'false',
  denylist: ['legacy-safari'],
}
这个开关只控制产品表虚拟滚动，不影响 workspace V2、users 表或其他 DataTable 调用方。
示例 5

开发态可观测性

<div
  data-virtual-enabled="true"
  data-virtual-count="18"
  data-virtual-total-size="112000"
  data-virtual-scroll-offset="3080"
  data-virtual-first-index="54"
  data-virtual-last-index="71"
/>
Playwright 和调试者都可以直接读这些属性，不再靠“看起来像是有渲染”这种模糊判断。
示例 6

降级事件队列

window.__DATA_TABLE_VIRTUAL_EVENTS__.push({
  type: 'fallback',
  reason: 'runtime-error',
  tableId: 'products-table',
  ts: Date.now(),
})
一旦线上触发局部降级，值班同学至少知道“何时、哪张表、为什么”，不会出现静默回退后没人察觉。

验收标准

这部分直接对应“上线前必须回答 Yes 的问题”。

量化指标

必须测出来的数字

  • DOM 预算 `perPage=2000` 时，产品表 `tbody tr` 总数应从 2000 降到 ≤ 64。
  • 行为正确 server pagination、排序、筛选变化时，不允许出现旧滚动位置先闪一帧。
  • 布局正确 sticky header 与右侧 pinned actions 列在中段滚动时仍完全贴齐。
硬性门槛

任何一条失败都不能宣布完成

  • fixed-height gate `cv` 和 `maxPrefixDrift` 不达标就停止该方案。
  • 可访问性 `aria-rowcount` / `aria-rowindex` 必须与当前页 row model 对齐。
  • 降级路径 配置开关、unsupported browser guard、runtime fallback 必须都能工作。

上线与回滚

企业商用不是“写出来”，而是“出了问题 5 分钟内能稳稳撤回”。

首发策略

只在 products 表试点

  • 范围控制 首发仅 products 表。users 表和其他 DataTable 调用方保持原样。
  • 兼容性矩阵 Chrome / Edge 当前与前一稳定版、Firefox 当前版、Safari 17.4+。
  • 辅助技术基线 至少手测 `VoiceOver + Safari` 与 `NVDA + Chrome`。
回滚策略

5 步撤退路径

  1. 关闭 `VITE_ENABLE_PRODUCT_TABLE_VIRTUALIZATION`。
  2. 重新部署，不动 workspace shell 其他逻辑。
  3. 验证 products 表恢复为完整 DOM 渲染。
  4. 检查事件队列中出现 `disabled-by-config` 或 `fallback`。
  5. 确认分页、筛选、排序行为仍正常。

结语

这不是一个“性能优化小补丁”，而是一套受控的渲染调度能力。

给架构师

边界清楚

只动 DataTable 渲染层，不把 `useDataTable`、feature 层查询状态和 workspace 结构搅在一起。

给前端

执行清楚

5 个任务顺序明确，先过方向 gate，再进入实现；每一步都有验证 profile，不会陷入“先改一坨再一起验”的失控状态。

给业务方

风险可控

首发只在 products 表，支持快速降级和回滚，出现异常时局部撤退，不牵连整页和整套 workspace。
