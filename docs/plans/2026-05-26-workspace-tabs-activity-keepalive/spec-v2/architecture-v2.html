Workspace Tabs V2 架构方案

轻壳 Tags + Activity 保活 + Feature-Local State — 2026-05-26 设计 / 2026-05-27 全部落地

一、为什么需要 V2？

V1 的问题
  • 太重：需要 WorkspaceRouteDefinition、searchAdapter、workspace-definition 三层抽象才能让一个页面进入 workspace
  • URL 绑架：表格的分页/筛选/排序状态必须和 URL search params 双向同步，造成不必要的复杂度
  • 耦合紧：adapter 把业务状态桥接到路由层，shell 理解了太多业务细节
  • 保活粒度错误：保的是"派生状态"而非"完整页面实例"
V2 的设计目标
  • 薄壳：页面只需要一个 WorkspacePageBoundary 组件即可接入
  • 内部状态：表格状态由 hook 内部维护，不再和 URL 绑定
  • 真正保活：用 React 19 <Activity> 保活整个页面实例，切换 tag 时状态自然保留
  • 职责清晰：shell 只管实例托管，feature 自己管业务状态

核心设计原则

V2 的四个核心理念构成了整个方案的骨架：

原则含义
薄壳（Thin Shell） WorkspaceShell 只做三件事：管理 tab 列表、托管页面实例（Activity）、协调关闭拦截（closeGuard）。不理解表格、搜索、分页等业务语义。
单宿主（Single Owner） flag-on 时，页面实例的唯一持有者是 ActivityHost。Route 只负责注册 descriptor 并返回 null，绝不允许 route 和 host 同时渲染同一个页面。
内部状态（Feature-Local State） 表格分页/筛选/排序是 table hook 的内部状态，表单草稿保存在页面子树内部。切换 tag 时因为 Activity 保活，这些状态自然保留，不需要额外的状态管理层。
描述符生命周期（Descriptor Lifetime） Route 的 mount/unmount 只表示"当前 URL 是否命中该 tab"，不影响 descriptor 的生死。Descriptor 注册后由 store 持有，只有 tab 被关闭或 shell 整体 reset 时才释放。

三、关键角色与职责

WorkspacePageBoundary

接入层

每个 dashboard route 的唯一接入点。它在 mount 时向 store 注册一个页面描述符（tabId、title、render 函数等），然后自己返回 null——真正的页面渲染完全交给 ActivityHost。

function ProductsRoute() {
  return (
    <WorkspacePageBoundary
      tabId="/dashboard/product"
      initialTitle="产品"
      render={() => <ProductsPage />}
    />
  );
}

ActivityHost

保活引擎

基于 React 19 <Activity>，是页面实例的唯一宿主。active 页面在 visible 模式渲染，hidden 页面保持实例不销毁。keepAlive=false 的页面在 deactive 后立即 unmount。

// active tab → 正常渲染
<Activity mode="visible">
  {descriptor.render()}
</Activity>

// hidden tab → 保活但不显示
<Activity mode="hidden">
  {descriptor.render()}
</Activity>

useWorkspacePage()

Lifecycle Hook

页面内部可选的 hook，用于在运行时更新 title、声明 dirty 状态、注册 closeGuard 关闭拦截器。Shell 不关心表单细节，只消费这个接口的结果。

const { updateLifecycle } = useWorkspacePage();

useEffect(() => {
  updateLifecycle({
    title: draft.name || '新建产品',
    dirty: isDraftDirty(draft),
    closeGuard: async () => {
      if (!dirty) return true;
      return await confirmDiscardChanges();
    },
  });
}, [draft, updateLifecycle]);

Workspace Store

状态中心

全局 store，持有所有 opened tabs 的 descriptor 和 lifecycle snapshot。它是 TagsBar 和 ActivityHost 之间的共享数据源。只存元信息，不存业务数据。

// Store 内部结构示意
{
  tabs: Map<string, {
    descriptor,      // 页面描述符
    lifecycle,       // { title, dirty, closeGuard }
    order,           // 打开顺序
  }>,
  activeTabId: string,
}

四、页面实例所有权时间线

这是 V2 最关键的约束——页面实例只能被 ActivityHost 渲染一次，不能出现双挂载。下面是一个典型的导航时序：

flag-on，用户导航到 /dashboard/product
  1. Route 计算出 tabId="/dashboard/product"、initialTitle="产品"、page component、keepAlive=true
  2. WorkspacePageBoundary 调用 register(descriptor) 写入 store
  3. WorkspacePageBoundary 返回 null（自己不渲染页面）
  4. WorkspaceViewport 在同一导航周期读取 active descriptor
  5. ActivityHost 唯一 mount 页面子树，渲染非空内容
用户切换到另一个 tag（如 /dashboard/users）
  1. TagsBar 更新 activeTabId → store
  2. ActivityHost 将 product 页面从 visible 切到 hidden mode
  3. ActivityHost 将 users 页面从 hidden 切到 visible mode
  4. product 页面实例不销毁，表格/表单状态完整保留
用户切回 /dashboard/product
  1. ActivityHost 将 product 页面从 hidden 切回 visible
  2. 之前的表格筛选、排序、分页状态都在，因为实例从未销毁
  3. 不需要 URL 恢复、不需要 adapter、不需要 route-state
keepAlive=false 的页面被 deactive
  1. ActivityHost 立即 unmount 该页面子树
  2. Route 不补直渲染（仍是 host 的职责）
  3. 下次激活时由 ActivityHost 重新 mount
用户关闭一个 tag
  1. 先检查该 tab 的 closeGuard（如果有）
  2. guard 通过 → store 清理 descriptor，ActivityHost 不再持有该实例
  3. guard 拒绝 → 中止关闭，焦点回到被拒绝的 tab
  4. 这是唯一允许触发 descriptor 清理的业务路径
用户执行 page.reload() 或 flag-off
  1. Shell 整体 reset，所有 descriptor 清空
  2. flag-off：回到无 tags / 无 Activity host 的单页模式
  3. 页面内部仍使用 v2 的 feature-local state，不复活 v1 的 adapter 链路

五、数据流对比：V1 vs V2

5.1 表格列表页的数据流

V1：URL 驱动的状态流
URL search params
↓ parse
searchAdapter / workspace-route-state
↓
useDataTable (外部状态)
↓
Table 渲染

问题：用户每次改筛选/排序都要改 URL，切换 tag 后再回来需要从 URL 重建状态。URL 变成了"通用状态存储"，带来不必要的复杂度。

V2：Feature-Local 内部状态
useDataTable (内部 state)
↓
DataTableController.state
↓ ↙ ↘
Table 渲染
Query 参数

表格状态由 hook 内部维护，切换 tag 时因 Activity 保活自然保留。URL 只做路由身份识别，不参与状态同步。

5.2 DataTableController 接口

V2 给 useDataTable 定义了清晰的"内部状态优先"契约：

type DataTableStateSnapshot = {
  pagination: { pageIndex: number; pageSize: number };
  sorting: SortingState;
  columnFilters: ColumnFiltersState;
};

type DataTableController = {
  state: DataTableStateSnapshot;           // 只读快照
  setPagination: (updater) => void;   // 修改分页
  setSorting: (updater) => void;      // 修改排序
  setColumnFilters: (updater) => void; // 修改筛选
  reset: () => void;                  // 重置所有状态
};
  • Controller 由 hook 自己持有，page 只消费
  • perPage 只影响初始 pageSize（从 localStorage preference 读取），之后由 controller 自行维护
  • router search 不再作为默认状态输入，也不再默认回写 URL

六、关闭拦截算法（CloseGuard）

当页面有未保存内容时，关闭 tag 需要经过统一的确认流程：

三种关闭动作

动作目标范围行为
close-current 只包含当前 active tab 若 dirty=false 或无 closeGuard，直接关闭；否则 await closeGuard() 决定
close-other 排除 active tab 外的所有 tabs 按 TagsBar 从左到右执行 guard，任一拒绝就立即中止，焦点切回被拒绝 tab
close-all 所有 tabs（含 active tab） 同上，从左到右依次 guard

Guard 异常处理（确定性行为）

情况结果
guard 返回 true允许关闭
guard 返回 false拒绝关闭
guard throw / rejected promise视为拒绝
guard 超过 1500ms 未响应超时视为拒绝

七、多实例详情页

每个完整的路由路径对应一个独立的 tag 实例：

/dashboard/product/new
/dashboard/product/123
/dashboard/product/456

三个不同实例，彼此状态隔离

function ProductDetailRoute() {
  const { productId } = Route.useParams();

  return (
    <WorkspacePageBoundary
      tabId={`/dashboard/product/${productId}`}
      initialTitle={productId === 'new' ? '新建产品' : `产品 ${productId}`}
      render={() => <ProductDetailPage productId={productId} />}
    />
  );
}

默认所有页面 keepAlive=true。如果某个详情/表单页确实不适合保活（如第三方编辑器嵌入、已知内存问题），允许显式 keepAlive=false opt-out，但必须在 review 中说明原因和恢复条件。

八、Feature Flag 双路径

flag-on（V2 主路径）

  • TagsBar + ActivityHost + feature-local state
  • 页面实例由 ActivityHost 唯一托管
  • 切换 tag 时实例保活，状态自然保留
  • 关闭 tag 走统一的 closeGuard 流程

flag-off（回退路径）

  • 无 TagsBar、无 ActivityHost
  • Route 直接渲染页面（传统单页模式）
  • 页面仍使用 V2 的 feature-local state
  • 不复活 V1 的 route-state / adapter / definition

flag 单点通过 src/config/workspace-tabs.ts 读取，业务代码不允许直接读 import.meta.env。flag-off 必须是"真正的关闭"，不允许出现"UI 看起来关了但 store/sync hook 还在跑"的半开状态。

十、保活策略矩阵

路由类型默认策略允许例外首轮要求
概览 / 列表 / 只读内容 keepAlive=true 极少 默认纳入
详情 / 表单 / 编辑器 keepAlive=true 可按证据 opt-out 默认纳入
chat / notifications / 重异步页面 keepAlive=true 若隐藏态副作用不可接受可 opt-out 必须先做 inventory
第三方嵌入 / 已知重型 canvas keepAlive=true 高概率 opt-out 需要 review 结论后落地

默认全部 true 是产品决策。只有明确证据（内存暴增、持续轮询、第三方编辑器恢复缺陷）才能 opt-out。首轮落地后只有两个重定向路由显式 opt-out。

十一、落地后调用形态

简洁性验收标准：如果 V2 落地后的调用形态不能满足下面 3 条，说明设计仍然过重。

  1. 普通页面只需要 WorkspacePageBoundary
  2. 脏页面只额外调用 useWorkspacePage().updateLifecycle
  3. 列表页只关心 useDataTable + query，不需要知道 shell、adapter、definition、route-state 的存在

示例 A：普通页面

function OverviewRoute() {
  return (
    <WorkspacePageBoundary
      tabId='/dashboard/overview'
      initialTitle='仪表盘'
      render={() => <OverviewPage />}
    />
  );
}

不需要 WorkspaceRouteDefinition、不需要 searchAdapter、不需要 parse/stringify/buildHref。

示例 B：列表页

function ProductsPage() {
  const { table, controller } = useDataTable({...});

  const filters = {
    page: controller.state.pagination.pageIndex + 1,
    limit: controller.state.pagination.pageSize,
    sorting: controller.state.sorting,
    columnFilters: controller.state.columnFilters,
  };
  const { data } = useSuspenseQuery(productsQueryOptions(filters));
  // ... 渲染表格
}

状态源就是 controller.state，query 直接消费。URL 不参与表格状态管理。

十四、V2 首轮明确不做的事

🚫 浏览器刷新恢复

不恢复刷新前打开的 tags 列表，不跨刷新恢复页面内部状态。刷新后只有当前 URL 对应的页面。

🚫 LRU / Tab 驱逐

首轮不引入自动实例驱逐策略。Shell 只提供实例计数和调试观测，不做容量管理。

🚫 URL 通用状态框架

不为"极少数需要 URL 状态的页面"同时设计通用框架。如有需要，feature 内部局部封装。
